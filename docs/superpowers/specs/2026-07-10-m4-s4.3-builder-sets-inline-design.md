# M4 S4.3 — Wstawianie ćwiczeń + inline edycja serii

Data: 2026-07-10
Branch: `feat/m4-builder-sets-inline`
Poprzedzone przez: S4.2 (`PlanBuilderTree` — edytor tygodnie→dni→sekcje, scalone do `main`); S4.1 (warstwa danych, w tym już gotowe `use-exercises.ts`/`use-sets.ts`, dotąd nieużywane przez UI); M3 S3.2 (`components/exercise-command.tsx`, kontrolowany komponent wyszukiwarki — zaprojektowany właśnie pod ten moment, zob. decyzja §9.17).

## Zakres

**W zakresie:**
- Wstawianie ćwiczeń do sekcji przez `exercise-command` (wyszukiwarka z tagami, Enter dodaje).
- Inline edycja serii: zakres powtórzeń („8" lub „8–10"), ciężar, RPE, przerwa — wszystkie zawsze widoczne (styl arkusza kalkulacyjnego), zero modali.
- Dodawanie/usuwanie ćwiczeń i serii.
- Opcjonalna notatka trenera przy ćwiczeniu (inline, `InlineEditableText`).

**Poza zakresem (S4.4):**
- Duplikacja serii/dnia/tygodnia.
- Drag&drop, reorder przez uchwyt.
- Skróty klawiszowe (np. „D" — duplikuj serię, Etap 2 per architektura-wizualna §5.1).
- Zmiana `set_number` przez trenera ręcznie (tylko sekwencyjne wyświetlanie; renumeracja to terytorium dnd).

## Architektura: rewizja inwariantu z S4.2

Decyzja §9.19 (S4.2) mówiła: `PlanBuilderTree` przyjmuje wyłącznie `context: PlanContext`. Ta sesja to rewiduje: wyszukiwarka ćwiczeń potrzebuje danych `exercises`/`archivedExercises`/`tags`, a zgodnie z ustaloną konwencją CRUD tego projektu (§9.16 — bank ćwiczeń to odczyty Server Component, nie zasób TanStack Query), te dane muszą przyjść jako propsy z Server Component, nie być pobierane po stronie klienta wewnątrz drzewa.

`app/(trainer)/templates/[id]/page.tsx` woła dodatkowo `listExercises(false)`, `listExercises(true)`, `listTags()` — te same trzy wywołania, które `(trainer)/layout.tsx` już robi dla globalnego menu ⌘K (pewna duplikacja na granicy page/layout, spójna z istniejącym precedensem „refetch per nawigacja" z §9.17, nie nowy wzorzec) — i przekazuje w dół: `<PlanBuilderTree context={...} exercises={...} archivedExercises={...} tags={...} />`.

**Jedna współdzielona instancja `<ExerciseCommand>`, montowana raz w korzeniu `PlanBuilderTree`** — nie jedna na sekcję. Stan `{ open: boolean, targetSectionId: string | null }` mieszka w `PlanBuilderTree`. Każdy `SectionRow` dostaje pojedynczy callback `onAddExercise: (sectionId: string) => void`, przewleczony przez `WeekNode`→`DayNode` (callback, nie dane — zero przewlekania exercises/tags poniżej korzenia). Klik „Dodaj ćwiczenie" na sekcji woła `onAddExercise(section.id)`, co otwiera współdzielony picker; wybór ćwiczenia woła `useCreatePlanExercise(context).mutate({ section_id: targetSectionId, exercise_id, exercise_name })` i zamyka picker.

## Komponenty (nowe + modyfikowane)

- **`lib/domain/reps-range.ts`** (nowy, czysta funkcja, testowana) — `parseRepsRange(input: string): { reps_min: number; reps_max: number } | null` (akceptuje `"8"` → stałe 8/8, `"8-10"` lub `"8–10"` → zakres; odrzuca nienumeryczne, `min > max`, ≤0; zwraca `null` przy błędzie) oraz `formatRepsRange(min: number | null, max: number | null): string` (renderuje `"8"` gdy równe, `"8–10"` gdy zakres, `""` gdy oba `null` — en dash do wyświetlania, zgodnie z przykładami copy w architektura-wizualna).
- **`components/plan-builder/set-row.tsx`** (nowy) — wiersz zawsze-widocznych inputów: etykieta „Seria N" (nieedytowalna — wyłącznie sekwencyjne wyświetlanie), input tekstowy zakresu powtórzeń (parsowany na blur przez `parseRepsRange`; błędny input cofa się do ostatniej poprawnej sformatowanej wartości zamiast commitować), inputy liczbowe ciężar/RPE/przerwa (każdy opcjonalny, puste ↔ `null`), przycisk usuń (`confirm()`, wzorzec z S4.2).
- **`components/plan-builder/exercise-row.tsx`** (nowy) — nazwa ćwiczenia (zwykły tekst, **nieedytowalny** — to niezmienny snapshot z reguły domenowej, więc brak afordancji zmiany nazwy), opcjonalna notatka inline (`InlineEditableText`, pusty stan „+ Dodaj notatkę"), przycisk usuń (`confirm()`, kaskada na serie), lista `SetRow`, przycisk „Dodaj serię".
- **`components/plan-builder/section-row.tsx`** (modyfikacja) — dodaje przycisk „Dodaj ćwiczenie" (woła `onAddExercise(section.id)` przekazane z korzenia) i renderuje `ExerciseRow` dla każdego ćwiczenia sekcji (już posortowane po `position` przez `mapRawTreeToPlanWeeks` z S4.1).
- **`plan-builder-tree.tsx` / `week-node.tsx` / `day-node.tsx`** (modyfikacja) — przewleczenie nowych propsów (`exercises`/`archivedExercises`/`tags` TYLKO w korzeniu; callback `onAddExercise` przez całą drogę w dół) i zamontowanie jednej współdzielonej `<ExerciseCommand>` w korzeniu.

## Domyślne wartości nowej serii

„Dodaj serię" kopiuje z OSTATNIEJ serii ćwiczenia: `reps_min`/`reps_max`/`target_weight`/`target_rpe`/`rest_seconds` do nowego wiersza (tylko `set_number` rośnie — liczony jako `sets.length + 1`, ten sam wzorzec „nie odporny na luki, ale natychmiast edytowalny", zaakceptowany już dla placeholderów dni w S4.2). Pierwsza seria na ćwiczeniu startuje całkowicie pusta (nie ma z czego kopiować).

## Obsługa błędów

Ten sam wzorzec co S4.2: każdy `onError` mutacji pokazuje toast `sonner` („Nie udało się zapisać serii.", „Nie udało się dodać ćwiczenia." itd.); każdy przycisk „dodaj" dostaje `disabled={mutation.isPending}` (ten sam guard przed podwójnym kliknięciem); każde usunięcie przechodzi przez `confirm()` nazywający co zostanie zniszczone. Błędny input zakresu powtórzeń (np. „abc" albo „10-8") NIE woła mutacji update — po prostu cofa input do ostatniej poprawnej wartości, bez tosta (nigdy nie opuścił klienta).
