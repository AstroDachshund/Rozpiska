# M3 S3.2 — Wyszukiwarka ćwiczeń (Command ⌘K) — design (spec)

Data: 2026-07-08 · Status: zatwierdzony w brainstormingu (Piotr) · Milestone: **M3**, sesja **S3.2** (`feat/m3-exercise-search`)

Kanon zakresu: `docs/plan-claude-code.md §3` (S3.2) i `docs/architektura-wizualna.md §5.1` (Command przy dodawaniu ćwiczenia w kreatorze — M4). Ten spec rozstrzyga: gdzie komponent jest realnie podłączony w S3.2 (M4 jeszcze nie istnieje), model filtrowania po tagach i widok zarchiwizowanych wewnątrz palety.

## 1. Rozstrzygnięcia (z brainstormingu)

- **Globalny skrót ⌘K/Ctrl+K w obrębie `(trainer)`** (nie tylko na `/exercises`) — dostępny z każdego ekranu trenera już teraz, mimo że jedynym konsumentem `onSelect` w S3.2 jest przejście do edycji ćwiczenia. M4 doda drugiego konsumenta (wstawienie do sekcji planu) korzystającego z tego samego reużywalnego komponentu z innym `onSelect`.
- **„Widok zarchiwizowanych" = toggle wewnątrz palety** (nie tylko strona `/exercises?archived=1` z S3.1). Chip/przycisk „Pokaż zarchiwizowane" przełącza, którą z dwóch list (aktywne / zarchiwizowane) przeszukuje paleta — ten sam model mentalny co strona z S3.1, ale osiągalny bez opuszczania wyszukiwarki.
- **Filtr tagów: semantyka AND.** Wybór wielu tagów zawęża wynik (ćwiczenie musi mieć WSZYSTKIE zaznaczone tagi), nie rozszerza. Domyślne rozstrzygnięcie — najbardziej użyteczne przy szukaniu konkretnego wariantu ćwiczenia; do korekty, jeśli w użyciu okaże się zbyt restrykcyjne.
- **Podział komponentów** (granica pod przyszłe M4):
  - `components/exercise-command.tsx` — **kontrolowany**, reużywalny picker: propsy `open`, `onOpenChange`, `onSelect(exercise)`, `exercises`, `archivedExercises`, `tags`. Nie zarządza własnym stanem otwarcia ani skrótem klawiszowym — to sprawa wywołującego (M4 poda własny trigger/onSelect przy „Dodaj ćwiczenie" w sekcji).
  - `components/trainer-command-menu.tsx` (nowy, klient) — globalne okablowanie na dziś: nasłuch ⌘K/Ctrl+K w całym `(trainer)`, stan otwarcia, `onSelect` = nawigacja do `/exercises/[id]/edit` + zamknięcie.
  - Zamontowany w `app/(trainer)/layout.tsx` (RSC), który pobiera raz na nawigację: aktywne ćwiczenia (z tagami), zarchiwizowane ćwiczenia (z tagami), listę tagów. Bez nowej warstwy danych (TanStack Query świadomie poza zakresem do M4/M6 — decyzja §9.16).
  - Mały przycisk odkrywalności „Szukaj ćwiczeń (⌘K)" w layoutcie trenera (na razie brak stałego chrome/sidebar nawigacji — to przyszła praca UI, poza zakresem S3.2).

## 2. Filtrowanie — pure fn + cmdk

- `lib/domain/exercise-filter.ts`: `matchesTagFilter(tags: { id: string }[], selectedTagIds: string[]): boolean` — `true` gdy `selectedTagIds` jest puste, albo gdy każdy zaznaczony id występuje w `tags` (AND). Czysta funkcja, testy Vitest (puste, jeden tag, wiele tagów — wszystkie obecne / część brakuje, duplikaty).
- Tekstowe wyszukiwanie: wbudowany fuzzy filter `cmdk` na `CommandItem value={nazwa}` — bez własnej reimplementacji dopasowania tekstu.
- Renderowanie: lista przekazana do `Command` to `(showArchived ? archivedExercises : exercises).filter(e => matchesTagFilter(e.tags, selectedTagIds))`; `cmdk` dodatkowo zawęża po wpisanym query.

## 3. UI — `ExerciseCommand`

- `CommandDialog` (shadcn, Radix Dialog pod spodem — focus trap, Esc, już na liście dozwolonych komponentów bazowych).
- `CommandInput` (wyszukiwanie tekstowe, autofocus).
- Rząd chipów tagów nad/pod inputem, grupowany po kategorii (muscle_group / equipment / movement_pattern) — reużycie istniejącego `TagChip` (wariant `selectable`, już z `aria-pressed` i widocznym focusem z S3.1).
- Chip/przycisk „Pokaż zarchiwizowane" (toggle, wizualnie odróżnialny od tagów — nie jest tagiem domenowym).
- `CommandList` z `CommandEmpty` („Brak wyników.") i `CommandItem` per ćwiczenie (nazwa + małe chipy tagów jako podpowiedź, `onSelect` woła prop `onSelect(exercise)` i **nie** zamyka dialogu samodzielnie — zamknięcie to decyzja wywołującego, przez `onOpenChange`).
- Pełna klawiatura: strzałki/Enter z `cmdk` out-of-the-box; Tab dociera do chipów tagów i toggle archiwum (cmdk nie przechwytuje Tab).

## 4. `trainer-command-menu.tsx` (globalne okablowanie)

- Nasłuch `keydown` na `document`: `(e.metaKey || e.ctrlKey) && e.key === 'k'` → `preventDefault()` + toggle otwarcia. Sprzątanie listenera przy odmontowaniu.
- `onSelect`: `router.push(`/exercises/${exercise.id}/edit`)` + zamknięcie dialogu.
- Przycisk odkrywalności „Szukaj ćwiczeń (⌘K)" renderowany obok (stały, np. `fixed bottom-4 right-4` albo w rogu — bez budowania pełnego sidebar nav, który jest przyszłą pracą UI per architektura-wizualna §5.1).

## 5. `app/(trainer)/layout.tsx`

- Pozostaje RSC (async). Pobiera: `listExercises(false)` (aktywne), `listExercises(true)` (zarchiwizowane), `listTags()` — równolegle (`Promise.all`). Przekazuje do `<TrainerCommandMenu exercises=... archivedExercises=... tags=... />` obok `{children}`.
- Świadomie brak cache/staleness handling (małe zbiory per trener w MVP; TanStack Query wchodzi w M4/M6 — nie tutaj).

## 7. Poza zakresem S3.2

Wstawianie ćwiczenia do planu (M4 — inny `onSelect` na tym samym `ExerciseCommand`). Stały sidebar nawigacji trenera. Cache/offline wyszukiwarki. Fuzzy-search po notatce technicznej (tylko nazwa). Zmiana semantyki filtra tagów na OR-w-kategorii (można zrewidować po realnym użyciu).
