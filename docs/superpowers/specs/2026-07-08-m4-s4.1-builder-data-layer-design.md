# M4 S4.1 — Warstwa danych kreatora planów — design (spec)

Data: 2026-07-08 · Status: zatwierdzony w brainstormingu (Piotr) · Milestone: **M4**, sesja **S4.1** (`feat/m4-builder-data-layer`)

Kanon zakresu: `docs/plan-claude-code.md §4` (S4.1). Ten spec rozstrzyga: gdzie żyje `QueryClientProvider`, dokładny algorytm `position.ts`, kształt Zod per węzeł, granulację hooków mutacji i bezpieczeństwo hard-delete.

## 1. Rozstrzygnięcia (z brainstormingu)

- **Bez migracji.** Całe drzewo (`plan_templates`/`assigned_plans` + `plan_weeks/days/sections/exercises/sets`) + RLS + triggery dziedziczenia właściciela (`private.plan_weeks_inherit_ownership`, `private.plan_structure_inherit_ownership`, `public.plan_ownership_immutable`) istnieją od M1 (`20260705193046_plan_containers.sql`, `20260705193047_plan_structure.sql`). Ta sesja to WYŁĄCZNIE warstwa aplikacji: `lib/domain/position.ts`, `lib/plan-builder/*` (typy, Zod, odczyt drzewa, mutacje per węzeł), `QueryClientProvider`. **Zero komponentów UI.**
- **`QueryClientProvider` wchodzi TERAZ, w root layout** (`app/layout.tsx`), nie odkładany do S4.2. To infrastruktura (klient-komponent opakowujący `{children}`), nie widoczny UI — hooki z tej sesji są od razu w pełni funkcjonalne, a S4.2 nie musi domykać setupu providera.
- **Ownership kolumny nigdy nie idą z formularza.** Zod schemas per węzeł obejmują wyłącznie pola edytowalne + bezpośredni FK rodzica (np. `plan_days` bierze `week_id` + `name` + `position`) — `trainer_id`/`client_id`/`template_id`/`assigned_plan_id` NIGDY nie są wysyłane z klienta; triggery M1 wypełniają je z rodzica (dokładnie ten sam wzorzec co istniejące `supabase/tests/helpers.ts` — `insertReturningId('plan_days', { week_id, name })`).
- **Delete węzłów struktury = twardy DELETE, nie soft-delete.** `plan_weeks/days/sections/exercises/sets` NIE mają `archived_at` w schemacie (w przeciwieństwie do `exercises`/`plan_templates`/`assigned_plans`). Bezpieczne, bo `set_logs.plan_set_id` ma `ON DELETE SET NULL` — usunięcie węzła struktury nigdy nie kasuje historii, tylko odczepia referencję (§3.3 pkt 2: historia czyta wyłącznie zdenormalizowane dane `set_logs`).

## 2. `lib/domain/position.ts` — czysty algorytm pozycji

- `const STEP = 1000` — domyślny odstęp między rodzeństwem przy pierwszym wstawieniu/dołożeniu na końcu.
- `positionBetween(before: number | null, after: number | null): number | null`:
  - `before === null && after === null` → `STEP` (pierwszy element listy).
  - `before === null` → `after - STEP` (wstawienie na początek; jeśli `after - STEP` degeneruje się poniżej progu — patrz niżej).
  - `after === null` → `before + STEP` (dołożenie na końcu).
  - oba podane → `(before + after) / 2`, chyba że `after - before` jest mniejsze niż `MIN_GAP = 1e-7` → zwraca `null` (sygnał: potrzebny rebalans tej listy rodzeństwa).
- `rebalance(count: number): number[]` — świeże, równo rozstawione pozycje zachowujące kolejność: `[STEP, 2*STEP, 3*STEP, ...]` dla `count` elementów.
- Testy Vitest: wstawienie do pustej listy, dołożenie na końcu, wstawienie na początek, wstawienie między dwoma sąsiadami, powtarzane połowienie aż do degeneracji (`null`), `rebalance` dla różnych `count` (0, 1, wiele) daje rosnący, równo rozstawiony ciąg.

## 3. `lib/plan-builder/types.ts` + `schemas.ts`

- `type PlanContext = { kind: 'template'; templateId: string } | { kind: 'assigned'; assignedPlanId: string }` — jeden dyskryminowany typ używany wszędzie (odczyt + mutacje), zamiast dwóch równoległych rodzin hooków.
- Typy wierszy węzłów (`PlanWeek`, `PlanDay`, `PlanSection`, `PlanExercise`, `PlanSet`) — pola faktycznie potrzebne klientowi (bez nadmiarowych kolumn właścicielskich w typach używanych przez UI, choć wracają w surowym select).
- Zod input schemas (jedno źródło pod przyszłe formularze S4.2/S4.3):
  - `weekInputSchema`: `{ week_number: number (int, ≥1); notes?: string }`.
  - `dayInputSchema`: `{ week_id: uuid; name: string (trim, min 1, max 120) }`.
  - `sectionInputSchema`: `{ day_id: uuid; section_type: enum('warmup','main','cooldown') }`.
  - `planExerciseInputSchema`: `{ section_id: uuid; exercise_id: uuid; exercise_name: string (snapshot, min 1) ; trainer_note?: string }` — `exercise_name` skopiowane z wybranego ćwiczenia w momencie dodania (punkt integracji z `ExerciseCommand` z S3.2, wpięcie UI w S4.3).
  - `setInputSchema`: `{ plan_exercise_id: uuid; set_number: number (int, ≥1); reps_min?: number; reps_max?: number; target_weight?: number; target_rpe?: number; rest_seconds?: number }` z regułą `reps_max >= reps_min` gdy oba podane (lustrzane odbicie CHECK `plan_sets_reps_order`).
  - Testy Vitest: po jednym przypadku valid + kluczowe invalid (brak wymaganego pola, złe typy, `reps_max < reps_min`) na schemat.

## 4. Odczyt drzewa — `lib/plan-builder/queries.ts`

- `fetchPlanTree(context: PlanContext)` — jedno zapytanie Supabase z zagnieżdżonym select (`plan_weeks(*, plan_days(*, plan_sections(*, plan_exercises(*, plan_sets(*)))))`), filtrowane po `template_id` lub `assigned_plan_id` zależnie od `context.kind`, posortowane po `position` na każdym poziomie.
- `usePlanTree(context: PlanContext)` — `useQuery` z kluczem `['plan-tree', context]`.

## 5. Mutacje per węzeł — `lib/plan-builder/mutations/*`

Po jednym pliku na typ węzła (`use-weeks.ts`, `use-days.ts`, `use-sections.ts`, `use-exercises.ts`, `use-sets.ts`), każdy eksportuje `useCreateX`/`useUpdateX`/`useDeleteX`:
- **Create**: czyta obecne pozycje rodzeństwa z cache `usePlanTree` (`queryClient.getQueryData`), liczy nową pozycję przez `positionBetween` (domyślnie: dołożenie na końcu — `positionBetween(lastPosition, null)`), jeśli `null` (degeneracja) → najpierw `rebalance` całego rodzeństwa (batch update pozycji), potem wstawia z pozycją ze świeżego rebalansu.
- **Update**: aktualizacja pól edytowalnych (i opcjonalnie `position` — reorder, ale UI reorderu to S4.2/dnd to S4.4; hook wspiera update pozycji już teraz).
- **Delete**: twardy DELETE (bezpieczny — patrz §1).
- Każda mutacja invaliduje/refetchuje `['plan-tree', context]` po sukcesie (`onSettled`).

## 6. `QueryProvider`

- Nowy plik `components/query-provider.tsx` (klient), tworzy jeden `QueryClient` (przez `useState`, żeby przeżył re-rendery, nie re-montowanie) i opakowuje `{children}` w `app/layout.tsx`, nad grupami tras.

## 7. Testy i DoD

- Vitest: `lib/domain/position.ts` + wszystkie Zod schemas z §3.
- Hooki (`usePlanTree`, mutacje) BEZ testów jednostkowych — projekt nie ma jsdom/testing-library (ta sama granica co S3.2); poprawność hooków zweryfikuje S4.2, gdy UI zacznie ich realnie używać.
- Brak nowych testów RLS (istniejące `tenant-isolation.test.ts`/`ownership-triggers.test.ts` już pokrywają to drzewo).
- Brak migracji, brak `gen types` (schemat niezmieniony).
