**Goal:** A complete, tested data layer for the plan-builder tree (`plan_weeks → plan_days → plan_sections → plan_exercises → plan_sets`) — position algorithm, Zod schemas, a one-query tree read, and per-node CRUD mutation hooks — with zero UI, ready for S4.2+ to build against.

**Architecture:** The DB schema, RLS, and ownership-inheritance triggers already exist (M1). This session adds only the app-layer: `lib/domain/position.ts` (pure), `lib/plan-builder/{types,schemas,queries}.ts` + `lib/plan-builder/mutations/*` (TanStack Query hooks using the browser Supabase client), and a `QueryProvider` mounted at the root layout.

**Tech Stack:** Next.js 15 (client hooks) · `@tanstack/react-query` v5 · Supabase browser client (`@/lib/supabase/client`) · Zod · Vitest.

## Global Constraints

- **No migration, no `gen types`** — schema unchanged (`supabase/migrations/20260705193046_plan_containers.sql`, `20260705193047_plan_structure.sql`).
- **Zero UI in this PR** — no new routes, no new visual components. `components/query-provider.tsx` is infrastructure (no visual output), not a UI component.
- **Ownership columns never come from the client.** `trainer_id`/`client_id`/`template_id`/`assigned_plan_id` on `plan_days/sections/exercises/sets` are auto-derived by the M1 `BEFORE INSERT` triggers from the parent row — mutation hooks never set them (except `plan_weeks`, the tree root, which must supply exactly one of `template_id`/`assigned_plan_id` per `PlanContext`; `trainer_id`/`client_id` are still trigger-derived even there).
- **Known TypeScript gotcha, handle it the same way everywhere:** the generated `Database['public']['Tables'][...]['Insert']` type marks `trainer_id` (and, for child tables, `client_id`) as **required** (`string`, no `?`) because the DB column is `NOT NULL` — even though the trigger fills it before that constraint is checked. Build the insert payload with only the columns you actually send, then assert it as the generated `Insert` type:
  ```ts
  const payload = {
    week_id,
    name,
    position,
  } as Database['public']['Tables']['plan_days']['Insert'];
  ```
  This is safe and intentional — the trigger supplies the missing NOT NULL columns before the constraint is checked — not a type-safety hole. Use this pattern in every mutation file; do not invent a different workaround per file.
- **Position flow (`lib/domain/position.ts`, Task 1)**: every create-mutation appends at the end of its sibling list via `positionBetween(lastSiblingPosition, null)` (never hardcode `+ STEP` inline — always go through `positionBetween`). This call can never return `null` — the `after === null` branch resolves to `before + STEP` before the degeneracy check ever runs — so create-mutations use it directly (`positionBetween(lastPosition, null)!`) with no rebalance branch. `rebalance(count)` exists in `position.ts` and is tested there, but is reserved for a future between-siblings insert (real reorder/drag-and-drop, S4.2/S4.4) where `positionBetween(before, after)` is called with two real neighbors and CAN degenerate — it is correctly unused by this session's create-mutations (found and fixed during Task 5's review; do not reintroduce a `rebalance` call in Tasks 6/7).
- **Query key**: `planTreeQueryKey(context: PlanContext)` (Task 4) is the ONLY way any file constructs the plan-tree query key. Every mutation file imports it — never hand-rolls `['plan-tree', ...]` locally.
- **No Zod re-validation inside mutation hooks.** Validation happens once, at the form boundary (S4.2+, a system boundary); mutation hooks trust their typed input (internal boundary) — matches CLAUDE.md's "validate at system boundaries, trust internal code."
- **No component/hook-level tests** — this repo's `vitest.config.ts` runs with `environment: 'node'`, no jsdom/testing-library (same constraint noted in the S3.2 plan). Vitest covers only pure logic: `position.ts`, the Zod schemas, and the pure tree-shaping/lookup helpers in `queries.ts` (Task 4) — nothing that calls `useQuery`/`useMutation` directly.
- **No new RLS tests** — `supabase/tests/tenant-isolation.test.ts` and `ownership-triggers.test.ts` already cover this schema.
- Polish inline comments where they explain non-obvious *why* (matches existing codebase style); code identifiers in English.
- **Before every commit**: `npm run typecheck` → `npm run lint` → `npm run format:check` → `npm run test`. `format:check` is mandatory (CI fails on it).
- Branch `feat/m4-builder-data-layer` (already created, based on current `main` — S3.2 has not merged yet, but this session has zero file overlap with it). One PR.

// Algorytm pozycji rodzeństwa w drzewie kreatora (float między sąsiadami).
// Odstęp startowy STEP daje duży zapas przed degeneracją floatów przy
// wielokrotnym wstawianiu; gdy odstęp spadnie poniżej MIN_GAP, positionBetween
// zwraca null — sygnał, że caller musi najpierw zrebalansować całe rodzeństwo.
export const STEP = 1000;
const MIN_GAP = 1e-7;

export function positionBetween(before: number | null, after: number | null): number | null {
  if (before === null && after === null) return STEP;
  if (before === null) return after! - STEP;
  if (after === null) return before + STEP;
  if (after - before < MIN_GAP) return null;
  return (before + after) / 2;
}

/** Świeże, równo rozstawione pozycje (1000, 2000, ...) zachowujące kolejność. */
export function rebalance(count: number): number[] {
  return Array.from({ length: count }, (_, i) => STEP * (i + 1));
}
```

  week_number: z.number().int().min(1, 'Numer tygodnia musi być dodatni.'),

    .string()

  day_id: z.string().uuid(),

export type SectionInput = z.infer<typeof sectionInputSchema>;

export const planExerciseInputSchema = z.object({
  section_id: z.string().uuid(),
  exercise_id: z.string().uuid(),
  exercise_name: z.string().trim().min(1, 'Brak nazwy ćwiczenia.'),
  trainer_note: z
    .string()
    .trim()
    .max(2000, 'Notatka jest za długa.')
    .optional()
    .transform((v) => (v === '' ? undefined : v)),
});
export type PlanExerciseInput = z.infer<typeof planExerciseInputSchema>;

export const setInputSchema = z
  .object({
    plan_exercise_id: z.string().uuid(),
    set_number: z.number().int().min(1, 'Numer serii musi być dodatni.'),
    reps_min: z.number().int().positive().optional(),
    reps_max: z.number().int().positive().optional(),

type RawSet = Omit<PlanSet, never>;
type RawExercise = Omit<PlanExercise, 'sets'> & { plan_sets: RawSet[] | null };
type RawSection = Omit<PlanSection, 'exercises'> & { plan_exercises: RawExercise[] | null };
type RawDay = Omit<PlanDay, 'sections'> & { plan_sections: RawSection[] | null };
export type RawPlanWeek = Omit<PlanWeek, 'days'> & { plan_days: RawDay[] | null };

const byPosition = <T extends { position: number }>(a: T, b: T) => a.position - b.position;

/** Czyste mapowanie: surowy select PostgREST -> PlanWeek[] posortowane na każdym poziomie. */
export function mapRawTreeToPlanWeeks(raw: RawPlanWeek[]): PlanWeek[] {
  return [...raw]
    .sort(byPosition)
    .map((week) => ({
      id: week.id,
      week_number: week.week_number,
      notes: week.notes,
      position: week.position,
      days: [...(week.plan_days ?? [])].sort(byPosition).map((day) => ({
        id: day.id,
        name: day.name,
        position: day.position,
        sections: [...(day.plan_sections ?? [])].sort(byPosition).map((section) => ({
          id: section.id,
          section_type: section.section_type,
          position: section.position,
          exercises: [...(section.plan_exercises ?? [])].sort(byPosition).map((exercise) => ({
            id: exercise.id,
            exercise_id: exercise.exercise_id,
            exercise_name: exercise.exercise_name,
            trainer_note: exercise.trainer_note,
            superset_group: exercise.superset_group,
            position: exercise.position,
            sets: [...(exercise.plan_sets ?? [])].sort(byPosition),
          })),
        })),
      })),
    }));
}

export function getWeekDays(weeks: PlanWeek[], weekId: string): PlanDay[] {
  return weeks.find((w) => w.id === weekId)?.days ?? [];
}

export function getDaySections(weeks: PlanWeek[], dayId: string): PlanSection[] {
  for (const week of weeks) {
    const day = week.days.find((d) => d.id === dayId);
    if (day) return day.sections;
  }
  return [];
}

export function getSectionExercises(weeks: PlanWeek[], sectionId: string): PlanExercise[] {
  for (const week of weeks) {
    for (const day of week.days) {
      const section = day.sections.find((s) => s.id === sectionId);
      if (section) return section.exercises;
    }
  }
  return [];
}

export function getExerciseSets(weeks: PlanWeek[], exerciseId: string): PlanSet[] {
  for (const week of weeks) {
    for (const day of week.days) {
      for (const section of day.sections) {
        const exercise = section.exercises.find((e) => e.id === exerciseId);
        if (exercise) return exercise.sets;
      }
    }
  }
  return [];
}

export function planTreeQueryKey(context: PlanContext) {
  return ['plan-tree', context] as const;
}

const TREE_SELECT = `
  id, week_number, notes, position,
  plan_days (
    id, name, position,
    plan_sections (
      id, section_type, position,
      plan_exercises (
        id, exercise_id, exercise_name, trainer_note, superset_group, position,
        plan_sets (
          id, set_number, reps_min, reps_max, target_weight, target_rpe, rest_seconds, position
        )
      )
    )
  )
`;

export async function fetchPlanTree(context: PlanContext): Promise<PlanWeek[]> {
  const supabase = createClient();
  const column = context.kind === 'template' ? 'template_id' : 'assigned_plan_id';
  const value = context.kind === 'template' ? context.templateId : context.assignedPlanId;

  const { data, error } = await supabase.from('plan_weeks').select(TREE_SELECT).eq(column, value);
  if (error) throw error;
  return mapRawTreeToPlanWeeks((data ?? []) as unknown as RawPlanWeek[]);
}

export function usePlanTree(context: PlanContext) {
  return useQuery({
    queryKey: planTreeQueryKey(context),
    queryFn: () => fetchPlanTree(context),
  });
}
```

Keep this cast; do not try to make the Supabase client infer the 4-level-deep shape natively.

  const queryClient = useQueryClient();
  const supabase = createClient();

  return useMutation({
    mutationFn: async (input: WeekInput) => {
      const key = planTreeQueryKey(context);
      const tree = queryClient.getQueryData<PlanWeek[]>(key) ?? [];
